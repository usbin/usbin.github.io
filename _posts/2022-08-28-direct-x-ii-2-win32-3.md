    ---### 1. 마우스 좌표 받기
    
    Win32 프로그램의 생명주기는 WndProc안에서 이루어지므로 이 안에서 살펴볼 것이다.
    WndProc의 인자는 hWnd, message, wParam, lParam이 있는데 hWnd는 윈도우 핸들이라고 했고, message는 WM_COMMAND, WM_LBUTTONDOWN 등의 윈도우 메시지가 담겨 온다.
    wParam은 키보드가 눌렸을 때 키 정보가 들어오며, lParam은 마우스가 눌렸을 때 좌표가 담겨 온다.
    즉, lParam을 파싱하면 마우스 좌표가 된다.
    lParam은 LPARAM 타입인데, LPARAM을 참조해보면 LONG_PTR을 이름만 바꿔놓은 걸 알 수 있다. LONG_PTR을 참조하면 long타입으로 나온다.
    C++에서 long타입은 4byte다.
    즉, lParam은 4byte 타입 변수라는 소리다.
    
    그리고 마우스 좌표는 lParam의 상단 2byte, 하단 2byte에 담겨서 온다.
    하단 2byte는 x좌표, 상단 2byte는 y좌표다.
    2byte가 담을 수 있는 범위는 0~65535이므로 모니터 픽셀 좌표를 담기에 충분히 크기 때문에 2byte씩 담은 걸로 보인다.
    
    ```c++
    LOWORD(lParam) //x좌표
    HIWORD(lParam) //y좌표
    ```
    마우스 좌표는 이렇게 파싱할 수 있다.
    
    ### 2. 그리기
    
    먼저 그림을 그리는 과정을 생각해보면
    
    (1) 마우스가 클릭됐을 때 선을 그리기 시작한다 -> WM_LBUTTONDOWN
    (2) 마우스가 움직이는 동안 궤적을 따라 선을 그린다 -> WM_MOUSEMOVE
    (3) 마우스가 떼어졌을 때 그리기를 종료한다 -> WM_LBUTTONUP
    
    위와 같은 순서로 이루어질 거라는 걸 생각할 수 있다.
    
    셋다 윈도우 이벤트로 존재하므로, 각 이벤트가 들어왔을 때마다 처리를 해주면 된다.
    
    여기서 WM_MOUSEMOVE는 마우스가 움직이는 동안 계속 메시지가 들어온다. 그런데 우리는 좌클릭이 눌려있는 동안만 그려야 한다.
    "좌클릭이 눌려있는 중"이라는 상태는 따로 제공하는 변수가 없으므로 만들어야 한다.
    g_is_drawing이라는 bool타입 변수를 전역변수로 선언해주자.
    
    ```c++
    bool g_is_drawing = FALSE; //기본값 false
    LRESULT CALLBACK WndProc(HWND hWnd, UINT message, WPARAM wParam, LPARAM lParam)
    { 
    //...
    }
    ```
    
    또한, 선을 그리려면 시작점과 끝점이 있어야 한다.
    (그냥 좌클릭 한 상태로 마우스가 움직일 때마다 마우스 위치 픽셀을 색칠해주는 게 더 직관적인 것 같은데, GDI 함수에서 그런 기능은 제공하지 않는다.)
    따라서 좀전의 마우스 위치와 현재 마우스 위치를 잇는 선을 매번 그려주어야 한다.
    
    좌표를 저장할 MousePoint라는 구조체와 함께 g_prev_mouse_point를 선언해준다.
    
    (1) WM_LBUTTONDOWN 메시지가 들어왔을 때, g_prev_mouse_point에 마우스 좌클릭이 눌렸을 때 처음 위치를 저장한다. 그리고 g_drawing을 TRUE로 바꾼다.
    (2) WM_MOUSEMOVE 메시지가 들어왔을 때, g_drawing이 TRUE라면 g_prev_mouse_point를 시작점으로, 현재 위치를 끝점으로 하여 잇는 선을 그린다. 그리고 g_prev_mouse_point를 끝점 위치로 갱신한다.
    (3) WM_LBUTTONUP 메시지가 들어오면 g_drawing을 FALSE로 만든다.
    
    위의 과정을 코드로 작성하면 다음과 같다.
    
    ```c++
    struct MousePoint {
        WORD x;
        WORD y;
    };
    MousePoint g_prev_mouse_point;
    bool g_is_drawing = FALSE;
    LRESULT CALLBACK WndProc(HWND hWnd, UINT message, WPARAM wParam, LPARAM lParam)
    {
    
        switch (message)
        {
        case WM_LBUTTONDOWN: //마우스 왼클릭 누름 -> 그리기 시작
            {
                g_prev_mouse_point.x = LOWORD(lParam);
                g_prev_mouse_point.y = HIWORD(lParam);
                g_is_drawing = TRUE;
            }
            break;
        case WM_MOUSEMOVE: //마우스 왼클릭 눌림+이동 -> 그리기
            {
                if (g_is_drawing) {
                    MousePoint p;
                    p.x = LOWORD(lParam);
                    p.y = HIWORD(lParam);
                    HDC hdc = GetDC(hWnd);
                    MoveToEx(hdc, g_prev_mouse_point.x, g_prev_mouse_point.y, (LPPOINT)NULL);
                    LineTo(hdc, p.x, p.y);
                    g_prev_mouse_point = p;
                }
            }
            break;
        case WM_LBUTTONUP: //마우스 왼클릭 뗌 -> 그리기 종료
            g_is_drawing = FALSE;
            break;
        case WM_DESTROY:
            PostQuitMessage(0);
            break;
        default:
            return DefWindowProc(hWnd, message, wParam, lParam);
        }
        return 0;
    }
    ```

    ### 3. InvalidateRect()와 메시지 기반 렌더링의 문제점
    
    위의 예에선 마우스 입력이 들어올 때마다 동기적으로 그림을 그리기 때문에 필요성이 별로 느껴지지 않지만,
    게임에서 화면에 무언가를 그릴 땐 저런식으로 동기적으로 이루어지는 게 아니라 화면에 그려야 할 것들의 정보를 모두 저장해두고, 매 프레임마다 새로 그리게 된다.
    
    가령 게임에서 인벤토리를 열었다가 닫는다든가 하는 예시를 생각해보면, 인벤토리를 열었을 땐 그 위치에 인벤토리를 그리면 되지만, 닫는다면 뒤에 그려져있었을 배경을 다시 복원해주어야 한다.
    만약 인벤토리를 열 때마다 현재 위치의 그림을 저장했다가 닫을 때마다 복구하면 될 것 같지만, 이 경우 인벤토리를 연 상태에서 캐릭터가 움직이기라도 하면 인벤토리를 닫았을 때 배경이 이상하게 변해버리는 문제가 발생한다.
    따라서 가장 뒷 배경 -> 맵 타일 등 -> 캐릭터 -> 인벤토리 순서대로 매번 다시 그리게 된다.
    이걸 렌더링이라고 하는데, 아직 공부한 부분이 아니라서 나중에 포스팅으로 올릴 거고
    
    아무튼간 윈도우 메시지 중에는 WM_PAINT라는 메시지가 있다.
    렌더링 과정을 전부 WM_PAINT 메시지를 받았을 때 수행하도록 집어넣어놓고 필요할 때마다 WM_PAINT를 발생시켜서 그림을 그리는 식으로 사용할 수 있는데
    
    그럴 때 InvalidateRect(hWnd, 다시 그릴 좌표, TRUE)를 사용하면 WM_PAINT 메시지가 발생한다.
    
    마지막 인자를 FALSE로 넣으면 본래 화면을 남겨둔 채로 그리고, TRUE로 넣으면 본래 화면을 지운 채 그린다.(백지)
    
    
    
    문제는 이 경우 InvalidateRect를 수동으로 발생시킬 때마다 화면을 그리게 된다는 건데, 
    게임에선 원래 1초에 60번 정도 화면을 그린다.(60fps가 이것)
    그러니까 1/60초에 1번씩 주기적으로 화면을 다시 그린다는 거다.
    
    그럼 이 60fps를 구현하려면 InvalidateRect를 1/60초에 한 번씩 발생시키거나, 타이머를 사용해 1/60초마다 WM_TIMER 메시지를 발생시켜 이 메시지 처리구문에 렌더링을 집어넣을 수 있다.
    하지만 이 두가지 방식은 시간이 너무 오래 걸린다고 한다.
    윈도우 메시지를 발생 -> 메시지 큐에 집어넣음 -> 메시지 큐에서 꺼냄
    이라는 불필요한 과정을 거쳐야 하기 때문.
    
    따라서 윈도우 메시지를 사용하지 않고 렌더링을 하는 게 좋다고 한다.
    
    ```c++
    // 기본 메시지 루프입니다:
        while (GetMessage(&msg, nullptr, 0, 0))
        {
            if (!TranslateAccelerator(msg.hwnd, hAccelTable, &msg))
            {
                TranslateMessage(&msg);
                DispatchMessage(&msg);
            }
        }
    ```
    여기서 GetMessage는 메시지를 받지 않으면 멈추어있는다. 이걸 메시지가 들어오지 않아도 다음 구문으로 진행하도록 바꾼 다음,
    (1) 메시지가 들어왔을 때
    (2) 메시지가 들어오지 않았을 때
    두 가지 케이스로 나누어 처리하면 윈도우 메시지 큐를 통하지 않고도 주기적으로 렌더링이 수행되게 할 수 있다.

    ### 4. GetTickCount(), SetWindowText()
    
    두 함수는 그냥 새로 알게 된 함수라서 정리해두려 한다.
    
    GetTickCount()는 ms초를 반환하는데, 끝 지점에서 시작지점의 값을 빼서 몇 초가 흘렀는지 체크할 때 사용할 수 있다.
    SetWindowText(hWnd, 텍스트)는 창의 타이틀을 설정하는 데 사용한다.
