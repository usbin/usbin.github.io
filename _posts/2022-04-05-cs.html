정리

좋은 코드란?

읽기 쉬운 코드 -> 보편적인 '읽기 쉬운 코드'의 기준은 있으나 네이밍은 개개인에 따라 읽기 쉽게 느껴지는 기준이 다르므로 명확히 정의내리기 어려움.

주석을 잘 단 코드 -> 이상적이지만, 수시로 코드를 바꾸다보면 주석과 일치하지 않는 경우가 생기는 문제. 오히려 주석이 혼란을 야기할 가능성.

좋지 않은 코드가 생겨나는 이유?

1. 쓰이지 않는 코드를 과감히 삭제하기 두려움. 어디서 사용될지 모른다!

-> 코드의 물리적 거리가 멀기 때문 or 순수하지 않은 함수(입력값과 변하지 않는 값을 제외한 다른 변수를 사용하는 함수)가 side effect를 야기하기 때문

2. 응급처치를 한 코드. 로직을 추가할 때 기존 코드를 리팩토링 하지 않고 응급처치 했을 때.

-> 부채와 마찬가지. 무조건적으로 나쁜 방법은 아니지만, 적정선을 찾아야함. 이런 코드가 쌓이면 "좋지 않은 코드"에 가까워짐.

​

좋지 않은 코드를 줄이는 방법?

1. 추출이 아닌, 추상화.

중복을 제거할 때 단순히 코드의 공통점으로 추출하지 않고, 공통 기능을 중심으로 "추상화"를 해야 함. 중복되는 코드를 섣불리 추출했다가 차후에 양쪽에서 다른 기능을 제공해야 할 수도 있다는 것을 염두.

의존관계가 있는 것끼리 추상화를 하자.

class Pianist{ void Do() { //악기를 조율한다 //연주한다 } } class Guitarist{ void Do(){ //악기를 조율한다 //연주한다 } }

나쁜 예)

abstract class Player{ void Do() }

좋은 예)

abstract class Player{ void Adjust() void Play() }

​

​

2. 일관적인 네이밍을 하자.

onClickButton -> 어미 + 동작 + 대상

handleButtonClick -> 어미 + 대상 + 동작

위의 두 예는 네이밍 규칙이 다르다. 어떤 방법을 써도 무방하나 모든 코드에 있어서 일관적인 네이밍을 할 것.

3. 디렉터리를 잘 분리하자.

용도에 따라 분리하는 것보다 먼저 물리적 거리에 따라 분리 후, 세부 폴더로 용도에 따라 분리하는 게 낫다.

용도에 따라 분리하면 나중에 코드간 물리적 거리가 멀어져, 삭제하기 두려운 코드가 된다.

4. 확장성 있는 코드를 짜자.

그때그때 새 파라미터를 추가하지 말고, 최대한 확장성을 열어서 만들자.

5. 어쩔 수 없는 코드는 주석과 함께 격리.

불가피하게 복잡해진 코드는 주석을 달아 격리해서 제대로 관리하자.
