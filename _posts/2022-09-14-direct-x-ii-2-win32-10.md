    [9. friend 선언](#user-content-9-friend-선언)### 1. 텍스쳐란?
    텍스쳐란 오브젝트에 씌우는 이미지를 말한다.
    가령, 동그란 구 오브젝트에 농구공 텍스쳐를 씌우면 농구공처럼 보이고, 야구공 텍스쳐를 씌우면 야구공처럼 보이게 할 수 있는데
    이렇게 오브젝트의 겉모습과 질감 등을 표현하기 위해 씌우는 이미지를 말한다.
    
    그냥 이미지라고 불러도 되지만 게임 개발 용어로는 텍스쳐라는 단어가 더 정확한 표현인 것 같다.
    
    앞으로 프로젝트 폴더-content-texture 폴더에 텍스쳐를 모아둘 것이다.
    텍스쳐는 .bmp 확장자를 가진 비트맵 이미지를 사용할 것이다.
    
    ### 2. vs 경로 변경하기
    지금 프로젝트의 빌드 위치 및 실행 위치는
    디버그 모드로 실행했을 경우 프로젝트 폴더-x64-Debug
    릴리즈 모드로 실행했을 경우 프로젝트 폴더-x64-Release이다.
    
    > 빌드 위치: 빌드된 .exe 파일이 위치하는 곳. 더블클릭해서 실행할 수 있는 실행파일이 있음.
    > 실행 위치: vs에서 F5(디버깅) 또는 ctrl+F5(실행)으로 실행할 때 빌드 및 실행하는 폴더 위치
    
    앞으로 사용할 텍스쳐는 릴리즈 모드의 폴더 안에 넣어놓을 건데, 디버그 모드로 실행했을 때에도 릴리즈 폴더 안에서 가져오도록 할 것이다.
    
    따라서 폴더 구조를
    Output
    ├ bin
    │ ├ content
    │ │	└ texture
    │ │ └ player.bmp
    │ └ Program.exe
    └ bin_debug
       └ Program.exe	
    
    이렇게 만들고, bin은 릴리즈 모드, bin_debug는 디버그 모드의 폴더로 사용할 것이다.
    그리고 실행파일의 계층 깊이가 같으므로 한 번 폴더를 나간 다음 bin-content-texture-player.bmp 순서로 접근하게 만들 것이다.
    
    그러기 위해서 먼저 vs의 빌드 위치와 실행 위치를 바꿔준다.

    ### 3. 리소스 클래스
    
    모든 리소스가 상속할 리소스 클래스를 만들었다.
    
    ```c++
    //Res.h
    #pragma once
    class Res
    {
    private :
    	tstring key_;
    	tstring relative_path_;
    
    public:
    	Res();
    	virtual ~Res();
    	inline void set_key(const tstring& key) { key_ = key; };
    	inline void set_relative_path(const tstring& path) { relative_path_ = path; };
    	inline const tstring& get_relative_path() { return relative_path_; };
    
    };
    
    
    ```
    
    ```c++
    //Res.cpp
    #include "pch.h"
    #include "Res.h"
    
    Res::Res()
    	: key_(_T(""))
    	, relative_path_(_T(""))
    {
    }
    
    Res::~Res()
    {
    }
    
    ```
    
    리소스가 가져야 할 건 두 가지다.
    - 메모리에 올라온 리소스를 가져오기 위한 키값
    - 메모리에 올라오지 않은 리소스를 파일로부터 읽기 위한 상대경로
    
    이 상대경로는 bin폴더 안의 content 폴더를 기준으로 한다.
    가령, player.bmp는 "texture\\player.bmp"를 상대경로로 갖는다.
    
    프로그램이 어느 위치에 있든 상관없이 리소스를 다루기 위해 리소스는 상대경로를 쓰지만,
    실제로 파일을 읽어올 땐 절대경로로 읽어야 한다. 따라서 상대경로를 가져와서 절대경로로 바꿔주는 PathManager 클래스도 필요하다.
    
    ### 4. 텍스쳐 클래스
    
    리소스를 상속하는 텍스쳐 클래스이다.
    
    ```c++
    //Texture.h
    #pragma once
    #include "Res.h"
    #include "ResManager.h"
    
    class Texture : public Res
    {
    
    private:
    	HDC mem_hdc_;
    	HBITMAP hbitmap_;
    	BITMAP bit_info_;
    
    	Texture();
    	virtual ~Texture() override;
    
    public:
    	void Load();
    	inline UINT get_width() { return bit_info_.bmWidth; };
    	inline UINT get_height() { return bit_info_.bmHeight; };
    	inline HDC get_hdc() { return mem_hdc_; };
    	friend class ResManager;
    };
    
    
    ```
    ```c++
    //Texture.cpp
    #include "pch.h"
    #include "Texture.h"
    #include "PathManager.h"
    #include "Core.h"
    Texture::Texture()
    {
    }
    
    Texture::~Texture()
    {
    }
    void Texture::Load()
    {
    	tstring texture_path = PathManager::GetInstance()->GetContentPath() + get_relative_path();
    
    	mem_hdc_ = CreateCompatibleDC(Core::GetInstance()->get_main_hdc());
    	hbitmap_ = (HBITMAP)LoadImage(nullptr, texture_path.c_str(), IMAGE_BITMAP, 0, 0, LR_CREATEDIBSECTION | LR_LOADFROMFILE);
    	//옵션: DIB 섹션 비트맵으로 읽어오기, 두번째 인자를 파일 경로로 설정해서 직접 파일에서 읽어오기
    	// DDB방식: 리소스로 비트맵을 불러와 DIB -> DDB로 변환해 DC가 선택하는 방법
    	// DIB방식: 직접 파일 입출력을 통해 읽어와 DIB의 정보를 이용해 직접 픽셀을 찍어내는 방법
    	// DDB는 가볍지만 세부적인 픽셀 작업이 불가, DIB는 픽셀 제어 가능하지만 부하가 심함
    	// 이 둘을 섞은 중간 방식이 DIB 섹션이라고 함.
    	
    	GetObject(hbitmap_, sizeof(BITMAP), &bit_info_);
    
    	//메모리DC와 비트맵을 연결
    	HBITMAP org_bitmap = (HBITMAP)SelectObject(mem_hdc_, hbitmap_);
    	DeleteObject(org_bitmap);
    
    
    }
    
    ```
    
    텍스쳐 클래스가 가져야 할 건 세가지다.(리소스 클래스가 갖고 있는 항목을 제외하고)
    - 텍스쳐가 그려져있는 비트맵의 핸들(hbitmap)
    - 이 비트맵이 선택돼있는 메모리dc
    - 비트맵의 정보(가로 세로 픽셀 길이 등)
    
    그리고 텍스쳐 클래스가 제공하는 함수는 Load()함수가 있는데,
    Load() 함수는
    - 경로관리자(구현은 아래에 서술)를 사용해 자신의 상대경로를 절대경로로 변환한 다음, =>경로관리자 필요
    - 비트맵 파일을 읽어오고,
    - 화면DC(메인DC)를 가지고 새로 메모리DC를 생성한 다음, =>화면DC 핸들 필요
    - 이 메모리 DC에 비트맵을 선택해주고, 기존의 비트맵은 제거하고,
    - 비트맵의 정보를 내부 멤버변수로 저장한다.
    
    참고로 Load() 함수는 실행되기 전에 반드시 Res의 함수로 제공하는 set_key와 set_relative_path를 사용해 내부 멤버변수들이 초기화되어 있어야만 정상작동한다.
    또한 비트맵은 메모리DC에 먼저 그려놓은 다음 render 함수에서 비트복사 함수로 옮겨오도록 해야 하는데,
    메모리DC를 새로 생성하려면 화면 DC가 있어야 한다.
    따라서 Core 함수에서 화면DC를 가져오는 함수를 추가했다.
    
    ```c++
    //Core.h
    inline HDC get_main_hdc() { return hdc_; };
    ```
    
    
    
    ### 5. 경로 관리자 클래스
    
    위에서 사용한 PathManager의 구현이다.
    
    ```c++
    //PathManager.h
    #pragma once
    class PathManager
    {
    	SINGLETON(PathManager);
    private:
    	tstring content_path_;
    public:
    	void Init(); //content_path_ 초기화
    	const tstring& GetContentPath();	//content 폴더의 경로를 반환
    };
    
    
    ```
    
    ```c++
    //PathManager.cpp
    #include "pch.h"
    #include "PathManager.h"
    
    PathManager::PathManager()
    	: content_path_(_T("")) {
    
    }
    PathManager::~PathManager() {}
    
    void PathManager::Init()
    {
    	TCHAR current_dir[255] = _T("");
    	GetCurrentDirectory(255, current_dir);
    	//bin또는 bin_debug를 지우고 \bin\content\를 새로 붙임.
    	int len = _tcslen(current_dir);
    	for (int i = len - 1; i >= 0; i--) {
    		if (current_dir[i] == '\\') {
    			current_dir[i] = '\0';
    			break;
    		}
    	}
    	_tcsncat_s(current_dir, _T("\\bin\\content\\"), _tcslen(_T("\\bin\\content\\")));
    	content_path_ = tstring(current_dir);
    
    }
    
    const tstring& PathManager::GetContentPath()
    {
    	return content_path_;
    
    }
    
    ```
    
    반드시 프로그램이 시작했을 때 Init을 해주어야 한다.
    또한 리소스를 불러오는 모든 작업이 시작되기 전에 Init이 되어있어야 하므로, Core 함수에서 SceneManager보다 앞에서 Init() 함수를 실행해주었다.
    그리고 GetCurrentDirectory로 받아온 현재 폴더는 디버그 모드는 bin_debug, 릴리즈 모드는 bin 폴더로 끝나는데
    content 폴더는 bin 폴더에만 있으므로, 가장 마지막 폴더를 지우고 bin폴더부터 새로 경로를 붙여준다.
    
    
    ### 6. 리소스 관리자 클래스
    
    리소스를 로드하는 시점은 프로그램 내에서 처음으로 이 리소스가 요구되는 시점이다.
    그리고 리소스가 소멸되는 시점은 요구한 오브젝트가 소멸될 때로 할 수도 있지만, 한 번 로드한 텍스쳐는 어차피 복사해서 쓰기 때문에 여러 오브젝트가 공유해서 쓸 수 있다.
    따라서 사용한 오브젝트가 소멸됐다고 바로 소멸시키는 건 메모리 낭비다.
    
    따라서 리소스의 로드 시점은 똑같이 처음으로 이 리소스가 요구될 때로 잡고,
    리소스가 소멸되는 시점은 프로그램이 종료될 때 일괄적으로 소멸하도록 한다.
    
    그러기 위해서 리소스 관리자를 따로 만들어준다.
    
    
    ```c++
    //ResManager.h
    #pragma once
    
    class Texture;
    
    class ResManager
    {
    	//전체 텍스쳐 map<키, Texture*>로 관리
    	//텍스쳐 로드 요청받으면 (이미 로드돼있으면 로드 먼저 하고) 리턴
    	//소멸자에서 텍스쳐 모두 소멸
    	SINGLETON(ResManager);
    private:
    	std::map<tstring, Texture*> textures_;
    
    public:
    	Texture* LoadTexture(const tstring& key, const tstring& relative_path);
    };
    
    
    ```
    ```c++
    //ResManager.cpp
    #include "pch.h"
    #include "ResManager.h"
    #include "Texture.h"
    
    ResManager::ResManager() {}
    
    ResManager::~ResManager() {
        for (auto it : textures_) {
            delete it.second;
        }
        textures_.clear();
    }
    
    Texture* ResManager::LoadTexture(const tstring& key, const tstring& relative_path)
    {
        auto it = textures_.find(key);
        if (it != textures_.end()) return it->second;
    
        Texture* texture = new Texture();
        texture->set_key(key);
        texture->set_relative_path(relative_path);
        texture->Load();
        textures_.insert(std::make_pair(key, texture));
    
        return texture;
    
        
    }
    
    ```
    리소스의 로드 함수를 제공하고, 소멸될 때 소멸자에서 자신이 갖고있는 모든 리소스를 소멸시켜준다.
    로드함수는 이미 해당 키가 로드돼있으면 바로 리턴하고, 로드돼있지 않으면 로드 후 리턴해준다.
    
    또한 리소스는 map 자료구조로 저장해서 키와 Texture 포인터를 함께 저장하는 구조로 하는데,
    map 자료구조는 내부적으로 자동정렬되기 때문에 키값을 알고 있으면 O(log n)의 시간에 리소스를 찾을 수 있기 때문이다.
    또한 red black 트리구조로 되어있어 삽입과 삭제도 O(log n) 시간에 이루어진다. 
    
    벡터나 리스트 등의 자료구조를 사용하면 삽입은 O(1) 시간에 할 수 있지만 찾을 때 O(n)의 시간이 걸린다.
    리소스의 양이 방대해질수록 탐색 시간이 커지게 된다.
    
    이제 텍스쳐가 필요할 때 사용할 키 값과 상대경로만 넣어주면

    ### 7. 특정 픽셀만 제외하고 복사하기
    
    여기서 플레이어의 배경이 마젠타 색인데(진분홍색), RGB(255, 0, 255) 값으로, 투명한 픽셀에 사용하는 값이라고 한다.
    255, 0, 255 색상은 거의 사용되지 않아서 이 값을 그냥 투명한 색으로 사용하는 것이라고 한다.
    
    비트맵은 무조건 네모난 형태기 때문에 그냥 복사하면 게임 내의 모든 이미지는 네모배경을 반드시 갖게 된다.
    이건 너무 이상하기 때문에 이럴 때 배경을 255, 0, 255 색상으로 칠한 후 나중에 복사할 때 이 색상의 픽셀을 없는 걸로 치고 제외하는 방식을 쓴다.
    
    따라서 비트맵을 복사할 때 BitBlt 대신 255, 0, 255로 칠해진 픽셀만 제외하고 복사하는 함수를 쓴다.
    
    ```c++
    void Player::Render(HDC hdc)
    {
    	TransparentBlt(
    		hdc //목적지 dc
    		, get_rect().left //left 좌표
    		, get_rect().top //top 좌표
    		, get_rect().right - get_rect().left	//가로 길이
    		, get_rect().bottom - get_rect().top	//세로 길이
    		, texture_->get_hdc() //소스 dc(=비트맵이 선택된 메모리dc)
    		, 0 //비트맵의 left 좌표
    		, 0 //비트맵의 top 좌표
    		, texture_->get_width() //비트맵의 가로 길이
    		, texture_->get_height() //비트맵의 세로 길이
    		, RGB(255, 0, 255)); //무시할 픽셀 색상
    		
    }
    ```
    
    근데 이렇게 실행하면 링크 에러가 난다.
    
    혼자 했으면 한참 삽질했을 텐데, 강의에서 이유를 알려줬다.

    TransparentBlt의 도큐먼트를 보면 요구사항에 wingdi.h 헤더가 있다.
    이 헤더는 Windows.h 헤더에 include되어 있다고 한다.(include Windows.h라길래 Windows.h을 포함하고 있다는 뜻인 줄 알았는데 포함"되어" 있다는 뜻이었다...)
    지금 pch.h 헤더에 Windows.h를 이미 include했기 때문에, wingdi.h도 인클루드 되어 컴파일 에러는 나지 않은 것이다.
    하지만 함수 본체는 헤더파일에 없기 때문에 링킹 과정에서 에러가 난다(고 한다.)
    
    근데 그럼 그동안 필요한 함수 있을 때 .h 헤더 포함해서 함수 호출도 잘만 해왔는데 왜 갑자기 안 되는가 하는 의문이 든다.
    찾아보니 기본 api는 링킹할 때 헤더만 있어도 알아서 링킹되는데 TransparentBlt가 속한 wingdi.h는 기본 api가 아니기 때문에 에러가 나는 듯하다.
    
    따라서 사용하려면 헤더만이 아니라 라이브러리도 직접 명시해서 참조해야 한다.
    
    사진에서 보면 Msimg32.lib라는 라이브러리에 포함돼있음을 알 수 있다.
    
    pch.h 파일에 다음 한 줄을 추가해준다.
    
    ```c++
    //pch.h
    #pragma comment(lib, "Msimg32.lib")
    ```
    
    이렇게 하면 빌드가 잘 된다.
    
    
    ### 8. 메모리 누수 찾기
    
    추가적으로, 메모리 누수를 쉽게 알려주는 함수가 있다.
    
    ```c++
    //Core.cpp
    int Core::Init(HWND h_wnd, int width, int height) {
    	
    	//메모리 누수 체크
    	_CrtSetDbgFlag(_CRTDBG_ALLOC_MEM_DF | _CRTDBG_LEAK_CHECK_DF);
    	//...생략
    }
    ```
    
    위와 같이 선언해두면, 디버그 모드에서 프로그램을 실행하고 종료할 때 출력에 반환되지 않은 메모리를 알려준다.
    
    가령
    ```c++
    //Core.cpp
    int Core::Init(HWND h_wnd, int width, int height) {
    	
    	//메모리 누수 체크
    	_CrtSetDbgFlag(_CRTDBG_ALLOC_MEM_DF | _CRTDBG_LEAK_CHECK_DF);
        new int; //누수될 메모리
    	//...생략
    }
    ```

    ### 9. friend 선언
    
    아까 위에서 Texture 클래스의 하단에
    ```c++
    //Texture.h
    	friend class ResManager;
    
    ```
    이런 게 있었는데, Texture 클래스를 아무나 생성하지 못하게 생성자와 소멸자를 private으로 선언했기 때문에 원래대로면 ResManager에서 접근할 수가 없다.
    그렇다고 protected로 선언할 수도 없다. ResManager는 Texture를 상속하지 않기 때문.
    
    상속관계가 아닌 ResManager가 Texture 클래스를 생성할 수 있게 해주기 위해선 friend를 사용하면 된다.
    friend 선언을 해주면 private멤버 에 접근할 수 있다.
