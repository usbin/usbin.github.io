1. 접근하기

진짜 어렵다... 난이도는 골드4.
백준 골드수준, 프로그래머스 3단계 정도까진 풀 수 있어야 한댄다.
브론즈<실버<골드<플래티넘 순서로 어렵고, 5~1까지 숫자가 작을수록 어려우니,
골드4는... 반드시 풀어야 하는 문제인 셈이다.
더 어려운 골드3,2,1이 코테에서 기다리고 있을테니.

아무튼 이 문제는 BFS 응용 문제다.
사실 최단거리를 찾는 문제가 아니기 때문에 DFS를 써도 무방하다.
그래서 둘다 풀어보았다.

먼저 '이분그래프'라는 게 뭔지 찾아보니, 그래프를 두 집합으로 나눴을 때 같은 집합 안에는 인접 노드가 없도록 할 수 있는 그래프라고 한다.
이렇게 말해도 뭔 소린지 잘 이해가 안 된다.
우리가 이해할 수 있는 방식의 설명은 다음과 같다.

시작 노드에서부터 인접 노드를 탐색하며 번갈아가며 두 가지의 색을 칠할 때, 인접한 노드끼리 같은 색이 칠해진다면 이분그래프가 아니다. 

대충 이렇게 그래프가 연결돼있을 때, 인접 노드끼리 같은 색깔이 되지 않으므로 이 사진은 이분그래프이다.

그럼 이 문제에선 모든 노드의 색깔 정보를 저장해야 한다는 걸 알 수 있다. 또한 방문 여부도 함께 저장해야 하므로,
int marker[노드 개수+1]라는 배열을 선언해 -1일 땐 방문하지 않은 것으로, 0은 빨간색, 1은 파란색으로 칠해짐을 저장하도록 했다.

그리고 인접 행렬을 선언해야 하는데...
노드 개수+1 * 노드 개수+1 개의 2차원 bool 행렬을 선언해선 안 된다.

노드 2만개는 2만*4바이트니까 marker가 차지하는 건 8만바이트=80KB지만,
2만개 * 2만개는 20,000*20,000 = 400,000,000byte = 400MB다. 메모리제한 256MB를 훌쩍 넘는다.
따라서 인접행렬은 bool 2차원 배열로 하지 않고, 각 노드당 연결된 노드의 번호를 벡터로 저장하도록 했다.

std::vector<std::vector<int>> adjacent_v
이런 형태다.

bool Bfs(int start_node, int max_node, std::vector<std::vector<int>>& adjacent_v) {
 
    std::queue<int> q;
    q.push(start_node);
 
    marker[start_node] = RED;
 
    while (!q.empty()) {
        int cnode = q.front();
        int ccolor = marker[cnode];
 
 
        for (int next : adjacent_v[cnode]) {
            if (marker[next] == NOT_VISITED) {
                //인접노드에 아직 마크하지 않았을 경우
                if (ccolor == RED) {
                    marker[next] = BLUE;
                }
                else {
                    marker[next] = RED;
                }
                q.push(next);
            }
            else if (ccolor == marker[next]) {
                return true;
            }
        }
        q.pop();
    }
 
    return false;
}

14번째 줄처럼, 모든 노드에 대하여가 아니라 모든 인접 노드에 대해서만 for문을 돌리면 시간복잡도를 훨씬 줄일 수 있다.

대충 계산해봐도 2차원 bool 인접행렬을 썼을 때 Bfs의 시간복잡도는 while문에서 n번(노드 개수 n), for문에서 n번 돌게 되어 있어 O(n^2)인데, 2차원 간선벡터를 사용하면 while문에서 n번 돌면서 인접행렬이 있을 때만 for문이 실행되므로 O(n+k)이다. k는 간선의 개수.
O(nk)가 아니라 O(n+k)인 이유는 모든 노드에 대하여 모든 간선 개수만큼 for문이 도는 게 아니라 모든 노드에 대한 while문이 다 돌아도 for문은 결국 전체 간선 개수만큼만 돌았을 것이기 때문이다. 

인접행렬과 간선리스트(레전드 바보코딩)

사실 이 부분은... 이전에 풀었던 BFS에서도 썼으면 좋았을 방식인 것 같다.
인접행렬이 아니라 간선리스트를 쓰는 방식은 상상도 못 했다....

심지어 나는 이 문제로 삽질하는 동안 인접행렬에서 간선리스트로 개선한 후에도 (모든 노드에 대하여 '이 노드가 간선리스트에 있을 경우')를 for문으로 돌리는 만행을 저질렀다. 거기다 나름 검색의 시간복잡도를 줄인다고 검색은 BinearySearch로 구현했다. O(n)으로 할 수 있는 일을 굳이 O(nlogn)으로 하면서 최적화했다고 생각했던 것이다. 바보코딩 레전드다... 이러니 시간초과가 나지.

물론 미로찾기처럼 맵을 입력받는 경우엔 각 위치에서 간선 리스트로 변환시키는 과정이 들어가므로 적절치 못하지만, 간선리스트를 받을 경우엔 굳이 인접행렬 이딴 짓 하지 말고 얌전히 벡터에 넣도록 하자.



아무튼 Bfs 함수에서는 큐에서 하나를 꺼낸 후, 그 인접한 노드들을 순회하며 큐에서 꺼낸 노드와 다른 색으로 모두 칠하는데, 만약 이미 색이 칠해져있으면서 꺼낸 노드와 같은 색이라면 => 이분그래프가 아닌 것! 이다.
그러므로 바로 true를 리턴해버린다.
만약 끝까지 돌았다면 이분그래프의 요건을 위배하는 노드를 찾지 못한 것이므로 false를 리턴한다.

각 테스트 케이스마다 간선리스트를 입력받으므로 간선리스트 초기화를 해주고,

bool Dfs(int start_node, int max_node, std::vector<std::vector<int>>& adjacent_v) {
    std::stack<int> stack;
    stack.push(start_node);
    marker[start_node] = RED;
 
    while (!stack.empty()) {
        int current = stack.top();
 
        bool all_searched = true;
        for (int next : adjacent_v[current]) {
            if (marker[next] == NOT_VISITED) {
                if (marker[current] == RED) marker[next] = BLUE;
                else marker[next] = RED;
                stack.push(next);
                all_searched = false;
                break;
            }
            else if (marker[next] == marker[current]) {
                return true;
            }
        }
 
        //모든 자식을 탐색했다면
        if (all_searched) stack.pop();
 
    }
    return false;
}

queue를 stack으로 바꾸고, 
큐에 항목을 넣은 뒤에도 for문을 계속 이어갔던 큐와 달리 Dfs에선 스택에 자식항목을 넣었다면 해당 계층 탐색은 break해야 하고, 스택에서 제거할 때에도 해당 계층 탐색 중 '탐색하지 않은 자식'이 있었는지를 체크한 뒤 제거해야 한다.


여담으로,



위가 Dfs, 아래가 Bfs인데, 어째 Bfs가 더 빨리 찾는다.
Bfs가 공간복잡도를 희생해서 속도를 빠르게 한 거라던데,
Bfs의 시간복잡도는 O(n+k)지만 Dfs의 시간복잡도는 대강 계산해보니 O(n+k^2)인 것 같다. 왜냐면 현재 노드의 간선이 5개라면 Bfs는 5개를 자식을 추가하고 현재 노드를 큐에서 제거해 탐색을 종료하지만, Dfs는 아직 탐색하지 않은 1개의 자식만을 추가하고 스택엔 그대로 남겨두고 break하기 때문에 결국 1+2+3+4+5번의 for문을 돌릴 것이기 때문이다. 
근데 O(n+k^2)과 O(n+k)의 차이치곤 두 배의 시간이 차이나는데...이유는 잘 모르겠다.
아무튼 Bfs가 더 빠르단 사실.
