객체지향 프로그래밍에서 단일책임의 원칙 때문이다.
: 하나의 클래스는 하나의 책임만을 가지고, 그 이유로만 변경이 가해져야 한다.

Player 클래스가 있고,
움직임을 담당하는 PlayerController가 있다고 가정하자.


2개의 클래스는 모두 하나의 게임오브젝트에 부착될 것이다.
그리고 각자 Update() 함수도 갖고 있다.

이걸 구현하는 방법은 두가지가 있다.



1. 메인 클래스가 컴포넌트 클래스를 호출하는 방법 : 아래 방향으로의 의존성

[RequipredComponent(typeof(PlayerController))]
public class Player : MonoBehaviour
{
    public InputSystem Input;    //키입력값이 들어간다고 가정
    PlayerController _playerController;
    
    void Start(){
        _playerController = GetComponent<PlayerController>();
    }
    void Update(){
        // 이동
        Vector3 movement = Input.GetMovement();
        _playerController.Move(movement*Time.deltaTime);
        
        // 점프
        if(Input.IsJumpKeyPressed()){
            _playerController.Jump();
        }
 
    }
}
 
 
 
[RequipredComponent(typeof(Rigidbody))]
public class PlayerController : MonoBehaviour
{
    Rigidbody _playerRigidbody;
 
    void Start(){
        _playerRigidBody = GetComponent<Rigidbody>();
    }
    
    public void Move(Vector3 movement){
        _playerRigidbody.MovePosition(_playerRigidbody.transform.position + movement);
    }
    public void Jump(){
        //점프
        _playerRigidbody.AddForce(위쪽으로 가하는 힘);
    }
}

이 경우, Player가 컴포넌트의 레퍼런스를 가지고 있고,
동작이 일어나는 조건과 입력값을 가공해서 컴포넌트로 넘겨준다.

실제로 Player와 PlayerController는 같은 오브젝트에 부착되었으므로 둘다 컴포넌트지만,
Player의 로직을 세분화해서 "움직임"만을 PlayerController에게 담당시킨 것이므로
둘만 떼어놓고 봤을 때 메인 클래스는 Player, 컴포넌트 클래스는 PlayerController가 된다.

메인->서브 호출일 때의 특징

Player는 PlayerController가 반드시 붙어있음을 전제로 한다. 따라서 PlayerController에 의존도를 갖는다.
PlayerController는 누가 호출하든 입력값만 주면 동작을 한다. 따라서 메인 클래스를 향한 의존도가 없다. 다만 같은 오브젝트에 붙은 Rigidbody를 사용하므로 Rigidbody에 의존도가 있다.

장점은 컴포넌트 클래스는 의존성이 없으므로 다른 데에서 재사용할 수 있다는 것. 컴포넌트 클래스에선 메인 클래스의 구현을 신경쓸 필요가 없다. 가독성이 좋다. 프로그램의 흐름을 이해하기 쉽다.
단점은 서브 클래스가 늘어날 경우 가공과 호출 루틴이 Player에 다 들어가므로 Player의 코드가 좀 더러워진다는 것, 그리고 데이터 교환을 파라미터로 하기 때문에 필요한 데이터에 변경이 일어나면 파라미터를 일일이 바꿔주는 노가다가 발생한다는 것이다.

public class Player : MonoBehaviour
{
    public InputSystem Input;    //키입력값이 들어간다고 가정
 
}
 
 
[RequipredComponent(typeof(Player))]
[RequipredComponent(typeof(Rigidbody))]
public class PlayerController : MonoBehaviour
{
    Player _player;
    Rigidbody _playerRigidbody;
 
    void Start(){
        _player = GetComponent<Player>();
        _playerRigidBody = GetComponent<Rigidbody>();
    }
    void Update(){
        // 이동
        Vector3 movement = _player.Input.GetMovement();
        Move(movement*Time.deltaTime);
        
        // 점프
        if(_player.Input.IsJumpKeyPressed()){
            Jump();
        }
    }
    public void Move(Vector3 movement){
        _playerRigidbody.MovePosition(_playerRigidbody.transform.position + movement);
    }
    
    public void Jump(){
        //점프
        _playerRigidbody.AddForce(위쪽으로 가하는 힘);
    }
}

이 경우, Player 클래스가 깔끔해진다. 컴포넌트가 독립적으로 활동하므로 뭘 따로 할 필요가 없기 때문이다. 그래서 Player가 여러 기능에서 계속 사용해야 할 Input만 남기고 다 PlayerController로 몰아넣을 수 있다.


서브->메인 참조일 때의 특징

위와 반대로, Player는 의존성을 갖지 않는 대신 PlayerController가 Player를 향한 의존성을 갖는다.

장점. 메인 클래스의 코드가 간결해진다. 메인 클래스에서 컴포넌트 클래스의 구현과 동작을 신경쓰지 않아도 된다. 컴포넌트 클래스의 변경이 이루어져도 메인 클래스에 영향을 미치지 않는다. 뭔가 변경이 일어나서 필요한 데이터가 바뀌어도 컴포넌트 클래스는 _player를 참조하고 있으므로 public으로 선언된 모든 데이터에 자유롭게 접근할 수 있다.

단점. 코드를 이해하기 어렵다... 메인 클래스의 구현에 종속적이므로 메인 클래스가 변경될 때마다 영향을 받는다. 컴포넌트를 개발할 때 메인 클래스의 구현을 확인해야 한다.




어느 방식이 좋은가?


결론부터 말하자면 무조건적인 정답은 없는 것 같다.

위의 방법(메인->서브 호출)을 쓰는 게 가독성 면에서는 낫다. 코드의 흐름을 파악하기 쉽고 각 컴포넌트간 데이터 교환을 쉽게 할 수 있다.
하지만 만약 이 컴포넌트를 다른곳에서 재사용하려 하니 수정해야 할 게 생기고 필요한 파라미터가 늘어난다면, 플레이어에까지 수정이 가해져야 해서 노가다가 발생한다.

아래 방법(서브->메인 참조)을 쓰면 가독성은 떨어진다. 컴포넌트를 구현할 때 컴포넌트가 종속적인 메인 클래스의 프로퍼티나 함수를 접근해야 하므로 코드를 계속 확인해야 한다. 그리고 컴포넌트에서 접근하고 있는 메인 클래스의 프로퍼티/함수가 뭔지 모르므로 수정할 수 없어진다. 개발할 때 컴포넌트를 하나하나 만들어서 붙인 뒤에 메인 클래스는 가장 마지막에 완성되기 때문에 변경이 자주 일어나게 된다. 
(만약 메인 클래스가 특정 인터페이스를 상속받게 해서 컴포넌트가 인터페이스의 참조를 갖도록 하면 의존성을 갖는 항목을 특정할 수 있다.)


그동안 컴포넌트 클래스를 개발할 때 메인 클래스에 자유롭게 접근하며 개발하는 게 편해서 아래 방식을 썼는데, 프로그램 구조가 점점 복잡해져서 나중엔 꼬이는 상황이 발생했다.
메인 클래스는 개발을 진행하며 계속 새로운 기능이 추가되기 때문에 변경이 자주 일어난다. 그런데 아래 방식을 사용하면 개발을 진행하며 클래스에 변경이 일어날 때 모든 컴포넌트에게 가는 영향을 신경써야 한다.


대충 정리하자면,

1. 메인 클래스가 자주 변경되면 위의 방식, 컴포넌트가 자주 변경되면 아래 방식이 좋은 것 같다.
2. 또 컴포넌트 서로간에 공유되는 데이터가 많다면 위의 방식, 컴포넌트 서로간 데이터 교환을 하지 않는다면 아래 방식이 낫다.
3. 컴포넌트에서 메인클래스의 함수를 반드시 호출해야 한다면 어차피 파라미터로 메인클래스의 참조를 전달해야 하므로 그냥 아래 방식을 쓰는 게 낫다.
