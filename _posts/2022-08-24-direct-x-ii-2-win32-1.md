    # 2. Win32 프로그래밍
    
    먼저, Win32 프로그래밍이란 Win32 API를 사용해서 프로그램을 만드는 것을 말한다.
    그럼 Win32 API란 무엇이냐?
    윈도우 환경에서 응용프로그램은 하드웨어에 직접적으로 접근할 수 없고 운영체제가 제공하는 함수만을 가져다 쓸 수 있는데,
    이 함수들의 집합을 Win32 API라고 부른다.
    
    16비트 윈도우를 사용할 땐 Win16 API라고 불렀다는데 32비트 윈도우가 나오면서 Win32 API로 이름이 바뀌었다.
    그럼 64비트 윈도우를 사용하는 지금은 Win64 API라고 불러야 하지 않나 싶은데... 왜인지는 모르겠으나 2017년 무렵까지만 해도 비주얼 스튜디오에서는 여전히 "Win32 프로그램"이라는 이름으로 불렸다.
    다만 비주얼 스튜디오 2019 기준, "Windows 데스크톱 어플리케이션"이라고 이름이 바뀌었다. 그럼에도 헤더파일은 여전히 Win32Program.h라고 붙어있는 걸 볼 수 있다.
    
    > 2022년 현재 Win32 API의 입지
    
    현재는 시스템 프로그래머쯤 되지 않으면 Win32 API까지 사용하지는 않는다고 한다.
    (여기저기 찾아보니 공통적으로
    - 이제는 구시대적인 유물이다.
    - 윈도우의 구조 이해에 도움이 되니 어느정도 익히고 있는 게 좋다.
    - 달달 외울 필요x 필요할 때만 책에서 찾아서 쓴다(soen.kr 참조하라고 함)
    - 요즘은 빡세게 공부할 필요가 없다
    라고 입을 모아 말함)
    
    마이크로소프트 가이드를 읽어보니 Direct X를 공부하기 전에 Win32 API에 대한 학습이 필요하다고 했다.
    따라서 간단한 Win32 프로그램을 하나 만드는 게 이 소단원의 목표이다.### Win32 프로그램
    
    일단 Win32 프로그램의 구조를 익혀보자.
    
    Win32 프로그램의 엔트리는 main함수가 아니라 wWinMain 함수이다.
    ```c++
    int APIENTRY wWinMain(_In_ HINSTANCE hInstance,
                         _In_opt_ HINSTANCE hPrevInstance,
                         _In_ LPWSTR lpCmdLine,
                         _In_ int nCmdShow)
    {
        UNREFERENCED_PARAMETER(hPrevInstance);
        UNREFERENCED_PARAMETER(lpCmdLine);
    
        // TODO: 여기에 코드를 입력합니다.
    
        // 전역 문자열을 초기화합니다.
        LoadStringW(hInstance, IDS_APP_TITLE, szTitle, MAX_LOADSTRING);
        LoadStringW(hInstance, IDC_WIN32PROGRAM, szWindowClass, MAX_LOADSTRING);
        MyRegisterClass(hInstance);
    
        // 애플리케이션 초기화를 수행합니다:
        if (!InitInstance (hInstance, nCmdShow))
        {
            return FALSE;
        }
    
        HACCEL hAccelTable = LoadAccelerators(hInstance, MAKEINTRESOURCE(IDC_WIN32PROGRAM));
    
        MSG msg;
    
        // 기본 메시지 루프입니다:
        while (GetMessage(&msg, nullptr, 0, 0))
        {
            if (!TranslateAccelerator(msg.hwnd, hAccelTable, &msg))
            {
                TranslateMessage(&msg);
                DispatchMessage(&msg);
            }
        }
    
        return (int) msg.wParam;
    }
    ```
    
    구조를 살펴보면
    
    1. InitInstace()로 초기화
    2. while문에서 루프를 돌며 메시지 처리(무한반복)
    
    이런 식으로 프로그램이 돌아가는 걸 볼 수 있다.
    
    InitInstance 함수의 코드는 다음과 같은데
    
    ```c++
    BOOL InitInstance(HINSTANCE hInstance, int nCmdShow)
    {
       hInst = hInstance; // 인스턴스 핸들을 전역 변수에 저장합니다.
    
       HWND hWnd = CreateWindowW(szWindowClass, szTitle, WS_OVERLAPPEDWINDOW,
          CW_USEDEFAULT, 0, CW_USEDEFAULT, 0, nullptr, nullptr, hInstance, nullptr);
    
       if (!hWnd)
       {
          return FALSE;
       }
    
       ShowWindow(hWnd, nCmdShow);
       UpdateWindow(hWnd);
    
       return TRUE;
    }
    ```
    
    우리 눈에 보이는 윈도우 창을 만들고 초기화하는 역할을 한다.
    
    또한, 기본적으로 윈도우 프로그램은 생명주기 내내 메시지를 입력받고 이 메시지에 대한 처리를 하는 것으로 이루어져 있다.
    어떤 메시지가 발생하면 메시지 큐에 담기고, 이 큐는 루프를 돌며 선입선출 방식으로 하나씩 꺼내져 처리된다.
    가령, 마우스가 어느 위치에서 클릭됐다는 메시지나 키보드의 무슨 키가 눌렸다는 메시지 등이 이에 해당한다.
    
    ### 메시지의 구조
    메시지 구조는
    ```c++
    /*
     * Message structure
     */
    typedef struct tagMSG {
        HWND hwnd;
        UINT message;
        WPARAM wParam;
        LPARAM lParam;
        DWORD time;
        POINT pt;
    #ifdef _MAC
        DWORD lPrivate;
    #endif
    } MSG, *PMSG, NEAR *NPMSG, FAR *LPMSG;
    ```
    위와 같이 이루어져 있는데, 하나씩 살펴보면
    hwnd는 윈도우 핸들을 뜻하고, message는 번호로 매겨진 윈도우 메시지이며(https://wiki.winehq.org/List_Of_Windows_Messages에서 어떤 번호가 무슨 메시지를 뜻하는지 찾을 수 있다)

    가령, 사진을 참고하면 마우스 왼클릭의 경우 10진수로 513으로 나타내어지는 걸 알 수 있다.
    wParam과 lParam은 파라미터를 뜻하고, time은 시간, pt은 위치이다(메시지가 마우스 클릭이라면 클릭 위치가 이에 해당)
    wParam과 lParam은 어디에 쓰이는진 모르겠는데 아마 메시지 번호, 위치, 시간에 더해 더 필요한 파라미터가 있을 경우 쓰이지 않을까 한다.
    
    
    ### 핸들
    
    위에서도 언급했던 핸들이라는 건 윈도우 프로그램에서 어떤 대상에 붙여진 번호로, 32비트 정수로 이루어져 있다.
    예를 들어 파일 핸들 같은 것이 있다. 어떤 파일을 열면 해당 파일엔 핸들이 붙는데, 100개의 파일을 연다면 100개의 각기 다른 핸들이 존재하게 된다. 이 핸들을 받아와서 핸들이 가리키는 대상을 처리할 수 있도록 되어있다.
    그렇다고 핸들이 포인터인 것은 아니다. 그냥 단순한 숫자일 뿐이고, 사실 개발자는 핸들이 숫자인지 문자인지 알 필요가 없다.(하지만 사실 핸들은 다 정수값이라고 한다. 문자보다 숫자 처리가 빠르기 때문에...)
    아무튼 우리는 그냥 '핸들'을 발급받은 후에 이 핸들을 파라미터로 넘겨주며 처리할 수 있다는 것만 알면 된다. 주민등록번호 같은 개념이라고 이해하면 될 것 같다.
    다른 종류의 핸들끼리는 같은 값을 가질 수 있으나 같은 종류의 핸들끼리는 절대 중복된 값을 갖지 않는다.
    또한 핸들 변수는 접두어 h로 시작한다. hInstance, hwnd등등 모두 핸들이다.
    
    
    ### LPCWSTR, LPCSTR, LPSTR, 그리고 문자집합
    
    윈도우 프로그램에서 이 자료형을 자주 보게 되는데, 모두 \0으로 끝나는 c스타일의 문자열을 뜻하는 것이다.
    순서대로 long pointer const wide char\*, long pointer const char\*, long pointer char\*을 뜻한다.
    LP : long pointer
    C : const
    W : wide(unicode를 나타냄)
    STR : \0으로 끝나는 char형 배열
    
    문자 집합엔 멀티바이트와 유니코드가 있는데, 자료형에 w가 붙으면 이 중 유니코드를 뜻한다. 또한 L"Hello, world!" 같은 식으로 문자열 앞에 L을 붙여줘도 유니코드를 뜻한다.
    이것이 붙지 않으면 멀티바이트로 취급된다. 문제는 멀티바이트 집합은 각 언어마다 한 문자가 차지하는 게 1byte일 수도 있고 2byte일 수도 있다. 따라서 문자 처리에 귀찮은 문제가 생긴다. strlen으로 문자의 길이를 알 수가 없다는 점 등.
    
    #### 싱글바이트? 멀티바이트? 유니코드?
    싱글바이트 문자 집합(SBCS): 1byte로 문자를 표현하는 문자 집합. ascii가 여기에 속함.
    멀티바이트 문자 집합(MBCS): ascii 코드 + 다른 문자들을 포함한 문자 집합. ANSI, UTF-8이 여기에 속함. 각 언어마다 테이블이 달라서 다른 테이블로 디코딩하면 문자가 깨지는 문제가 있다. 또한 언어마다 차지하는 바이트 수가 다르다. 대표적으로 영어는 1byte, 한글은 2byte.
    유니코드(WBCS): 2byte 문자집합. 테이블이 단 하나뿐이라서 디코딩 문제가 없다. 바이트 수가 2바이트로 고정이라 문자를 다루기 편하지만 싱글바이트에 비해 차지하는 메모리가 두 배가 된다는 문제. 또한 함수 사용을 위해 유니코드용 등 새로 정의된 다른 함수를 사용해야 함.
    
    ### tchar.h
    tchar.h 헤더를 사용하면 MBCS인지 WBCS인지 신경쓰지 않고 코딩을 할 수 있는 함수들을 사용할 수 있다.
    자료형과 함수들이 들어있음.
    참조: https://home.cs.colorado.edu/~main/cs1300-old/include/tchar.h
    
    <자료형>
    CHAR 또는 WCHAR -> TCHAR (char)
    LPSTR 또는 LPWSTR -> LPTSTR (char*)
    LPCSTR 또는 LPCWSTR -> LPCTSTR (const char*)
    
    <사용할 때>
    ```c++
    TCHAR str[] = _T("Hello!");
    _tprintf(str);
    ```
    
    이렇게 t 접두어를 붙인 tchar.h의 함수와 자료형을 사용하면, 
    ```c++
    #define _UNICODE
    ```
    를 프로그램 상단에 넣어주는 것만으로 유니코드로 컴파일이 된다. 넣어주지 않으면 디폴트로 MBCS 컴파일이 된다.
    
    _trpintf보다 printf가 익숙하지만 호환성을 위해 앞으로는 습관적으로 tchar.h 헤더를 사용하는 게 좋을 듯하다.
