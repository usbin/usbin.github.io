    # 1. ATL 라이브러리 
    (참고: https://docs.microsoft.com/ko-kr/cpp/atl/atl-com-desktop-components?view=msvc-170)
    ### (1) COM(구성 요소 개체 모델)
    COM은 상호작용할 수 있는 이진 소프트웨어 구성 요소를 만들기 위한 플랫폼 독립적이고 분산된 개체 지향 시스템.
    다양한 프로그래밍 언어로 만들 수 있지만 C++ 같은 개체 지향 언어는 COM 개체의 구현을 간소화하는 프로그래밍 메커니즘을 제공함.
    COM의 유일한 언어 요구사항은 코드가 포인터 구조를 만들고 포인터를 통해 함수를 호출할 수 있는 언어로 생성된다는 것.
    
    > 이진 소프트웨어라는 건 뭔지 모르겠으나 일단 '이진 표준'이라는 건 프로그램이 이진 머신 코드로 변환된 후에 적용되는 표준이라고 함.
    > 이진 소프트웨어도 같은 맥락에서 저수준 소프트웨어를 말하는 것 같음.
    
    일반적으로 소프트웨어 개체는 데이터 집합과 데이터를 조작하는 함수로 구성됨.
    COM 개체는 하나 이상의 관련 함수 집합을 통해서만 개체의 데이터에 액세스할 수 있는 개체. 이 함수 집합이 '인터페이스'이며 인터페이스의 함수가 '메서드'임.
    또한 인터페이스에 대한 포인터를 통해서만 인터페이스 메서드에 액세스할 수 있어야함.#### COM 인터페이스
    - COM 개체는 멤버 함수 컬렉션인 '인터페이스'를 통해 기능을 노출함.
    - 'COM 인터페이스'는 구성 요소의 예상 동작 및 책임을 정의, 작은 관련 작업 집합을 제공.
    - 모든 인터페이스에는 IID라는 고유한 인터페이스 식별자가 있음. 새 인터페이스를 만들 때 IID를 만들어야 함.
    - 단일 상속만 지원.
    
    C++에서의 인터페이스: 클래스가 지원하고, 개체의 클라이언트가 호출하여 상호작용할 수 있는 모든 함수
    COM 인터페이스: COM 클래스가 구현하는 미리 정의된 관련 함수 그룹을 참조. 하지만 반드시 클래스가 지원하는 모든 함수를 나타내는 것은 아님.
    
    ***인터페이스를 구현하는 개체를 참조한다*** 는 건 어떤 인터페이스를 따르는 개체가 COM 라이브러리에 해당 함수에 대한 COM 이진 규격 포인터를 제공하고, 프로세스 내부에 있든 외부에 있든 상관없이 포인터를 요청하는 모든 클라이언트가 해당 함수를 사용할 수 있도록 하는 것이다.
    
    #### IUnknown 인터페이스
    모든 COM 인터페이스는 IUnknown 인터페이스에서 상속됨.
    IUnknown 인터페이스에는 다형성 및 인스턴스 수명 관리를 위한 기본 COM 작업이 포함되어 있음.
    - QueryInterface, AddRef 및 Release라는 세 개의 멤버함수를 가짐.
    	- QueryInterface : 런타임에 호출하면 COM 개체가 특정 인터페이스를 지원하는지 여부 확인. 구현하는 경우 매개변수에서 인터페이스 포인터를 반환하고 그렇지 않으면 NULL 반환.
        - AddRef: 참조 개수 증가
        - Release: 참조 개수 감소
        (COM 개체 인스턴스의 수명은 참조 횟수에 의해 제어됨. 참조 수가 0에 도달하면 Release 함수가 인스턴스를 해제할 수 있음.)  
        
        
    #### 클라이언트/서버 모델
    COM 개체와 호출자 간의 상호작용은 클라이언트/서버 관계로 모델링 됨. 클라이언트는 시스템에서 COM 개체를 요청하는 호출자이며 서버는 COM 개체를 보관하는 모듈.
    COM 클라이언트는 CLSID를 시스템에 전달함.(CoCreateInstance 호출)
    - CoCreateInstance 함수: 지정된 CLSID의 인스턴스를 하나 만들고 요청한 형식의 인터페이스 포인터를 반환함.
    - 클라이언트는 사용을 마쳤을 때 Release 함수를 호출하여 인스턴스의 수명을 관리해야 함.
    - GetActiveObject 함수: 이미 만들어 실행중인 개체에 연결
    
    COM 서버는 CLSID를 COM 클래스와 연결하고, 클래스의 구현을 보관하고, 클래스 인스턴스를 만들기 위한 클래스 팩토리를 구현하며, 서버 언로드를 제공함.(COM 서버와 COM 개체는 동일하지 않음)
    - COM 서버에서 IClassFactory 인터페이스의 구현을 제공해야 COM 개체를 만들 수 있음.
    
    너무 길어지므로 자세한 내용은 링크...[https://docs.microsoft.com/ko-kr/windows/win32/com/com-technical-overview](https://docs.microsoft.com/ko-kr/windows/win32/com/com-technical-overview)
    #### 참조 개수
    COM은 개체가 더이상 사용되지 않아도 개체를 제거하지 않음. 프로그래머가 제거해야함
    
    #### 마샬링
    한 프로세스의 개체에 의해 노출되는 인터페이스를 다른 프로세스에서 사용할 수 있게 함.
    메서드의 매개변수를 프로세스간에 이동할 수 있는 형식으로 압축하고 다른쪽 단말에 도착하면 매개변수의 압축을 풀음.
    - 일반적으로 동일한 프로세스에서 인터페이스가 사용되는 경우 마샬링이 필요하지 않음. 그러나 스레드 간에는 필요할 수도 있음.
    
    #### 집계
    미리 빌드된 다른 개체에서 제공하는 서비스를 사용하는 경우, 또한 이 다른 개체를 개체의 일부분으로 자연스럽게 표시하려고 할 때.
    외부 개체가 내부 개체를 생성 프로세스의 일부로 만들고, 내부 개체의 인터페이스가 외부에 의해 노출되게 함.
    
    ### (2) ATL(활성 템플릿 라이브러리)
    
    ATL이란 COM 개체를 쉽게 만들 수 있는 템플릿 기반 C++ 클래스 집합.
    
    > Keywords : IUnknown, IClassFactory, IClassFactory2, IDispatch 이중 인터페이스의 스톡 구현, 표준 COM 열거자 인터페이스, 연결점, 분리 인터페이스, ActiveX 컨트롤
    
    단일 스레드 개체, 아파트 모델 개체, 자유 스레드 모델 개체 또는 자유 스레드 및 아파트 모델 개체를 만들 수 있음.
    
    > 템플릿이란? 매크로와 비슷함. 템플릿을 호출하면 작성한 코드로 확장됨. 다만 매개변수로 전달하는 형식에 따라 좀더 다양하게 확장됨.
    
    ATL은 소스코드로만 제공되며 본질적으로 계층적이 아니라는 점에서 기존 C++ 클래스 라이브러리와 다르다.
    클래스에서 파생되는 게 아니라 템플릿에서 클래스를 인스턴스화함.
    
    #### ATL과 MFC(Microsoft Foundation 클래스 라이브러리)
    
    ATL은 C++에서 COM 구성 요소를 만들고 작은 공간을 유지하는 빠르고 쉬운 방법.
    MFC를 사용하면 전체 애플리케이션, ActiveX 컨트롤 및 활성 문서를 만들 수 있음. ATL보다 제공하는 것이 많음.
