 객체지향 프로그래밍(OOP:Object Oriented Programming)이란?

현실 세계의 사물을 프로그래밍 세계로 가져와 적용하는 것. 인간 중심적 프로그래밍 패러다임.

(객체지향이 나오기 전에는 절차지향 프로그래밍이 있었는데, 컴퓨터가 사고하는대로 프로그래밍하는 패러다임이었음.)

​

프로그램을 객체와 객체간의 메시지 교환으로 구현한다.

​

메시지 교환이란?

-> 객체의 메서드를 호출하는 것.

Pen.Draw("Circle"); //객체 Pen에 Draw라는 메시지를 파라미터 "Circle"과 함께 보낸다.

​

장점:

한 번 클래스를 만들어두면 가져다 쓸 수 있으므로 재사용성이 증가한다.

데이터 모델링을 할 때 객체와 매핑하기 수월해진다.

디버깅이 쉬워진다. <- 객체에 접촉할 때마다 객체의 상태가 달라지므로 디버깅이 어려워질 것 같은데... 코드가 객체 단위로 나뉘어져서 디버깅이 쉬워진다고 써있다. 객체를 서로 독립적으로 설계하기 때문인 것 같다.

유지보수가 용이하다. (각 객체가 독립적으로 작동하므로)

​

단점:

객체간의 정보 교환이 메시지 교환에 의존하므로 오버헤드가 크다. -> 현대의 하드웨어의 성능 향상으로 보완된 문제.

객체가 상태를 가지므로 예측할 수 없는 버그를 발생시킨다.

​

객체지향 프로그래밍의 특징

인터페이스는 객체가 이해할 수 있는 메시지의 목록을 정의한다. 바꿔말하면, 메시지의 목록을 정의하기 위해 인터페이스를 작성해야 한다.

Person, Bar, Zym라는 객체가 있을 때, Bar에서 Person은 Drink를 하고, Zym에서 Person은 Exercise를 한다.

class Bar{ public void SomeFunc(Person person){ person.Drink(); } } class Zym{ public void SomeFunc2(Person person){ person.Exercise(); } }

이런 코드를 작성하고 싶다면, person이 Drink와 Exercise를 할 수 있다는 걸 인터페이스로 정의해야 한다.

Drink를 메시지의 목록으로 갖는 인터페이스는 직관적으로 그냥 IDrinkable,

Exercise는 IExercisable이라고 이름을 붙이자.

class Person : IDrinkable, IExercisable{ public void Drink(){ //마시기 } public void Exercise(){ //운동하기 } } interface IDrinkable{ public void Drink(); } interface IExercisable{ public void Exercise(); }

Person은 인터페이스를 상속받음으로써 내부에 Drink와 Exercise라는 함수를 반드시 구현해야 하고, Bar와 Zym은 확정적으로 Person이 Drink와 Exercise라는 메시지를 받을 수 있다는 사실을 알고 메시지를 보낼 수 있다.

​

객체지향 프로그래밍 기법

1. 객체와 객체를 좀더 추상적으로, 느슨하게 결합하기 -> 응집도 상승, 결합도 하락, 캡슐화 증진, 인터페이스와 구현의 분리, 설계의 유연성과 재사용성.

인터페이스를 정의할 땐 최대한 유연하게, 느슨하게 정의한다.

​

2. 상속은 기능의 확장(is-a 관계)에서만 사용하고, has-a 관계에선 참조를 사용하자.

상속 관계에서 상위 클래스를 변경하면 하위 클래스 전체가 영향을 받는다. 또한 상속을 사용하는 만큼 클래스 수가 늘어나게 되므로, is-a 관계일 때만 상속을 사용하자.

​

3. 객체보다 메시지에 집중해서 설계하자.

객체는 객체가 갖고 있는 데이터보다 어떤 메시지를 수신할 수 있느냐가 중요하다.

​

​

데미테르 법칙

1. 메서드에서 생성한 객체의 메서드만 호출.

2. 파라미터로 받은 객체의 메서드만 호출.

3. 필드로 참조하는 객체의 메서드만 호출.

​

​

객체지향의 특징

1. 캡슐화

접근 제한 지정자(public, private)를 사용해 사용자가 알아야 할 정보만 노출하고 나머지는 숨기는 것.

2. 상속

공통된 기능을 묶어 부모 클래스로 빼내어 재사용하는 것.

3. 추상화 (인터페이스, 추상클래스)

모델링이라고도 볼 수 있음. 추상화를 거친 결과물이 '클래스'.

의존성을 주입하기 위해 먼저 공통적인 특징, 변하지 않는 특성을 뽑아내 모델링한다.

인터페이스, 추상클래스 기능을 이용해 수신 가능한 메시지의 목록을 정의한다는 관점에서 캡슐화와도 관련이 있음.

부모 클래스는 메시지의 목록을 제공할 뿐이고 독립적인 객체로 사용되지 않는다. 자식 클래스만 사용된다.

​

*추상 클래스란?

- 말하자면 미완성 설계도

- 추상 멤버 함수 또는 추상 프로퍼티를 최소 1개 이상 가진 클래스를 말한다.(c# 기준)

- 일반 멤버함수 또는 멤버 변수, 프로퍼티도 포함할 수 있다.​

- 추상 클래스를 상속하는 클래스는 다른 클래스 또는 추상 클래스를 상속할 수 없다.

- IS-A 관계에서 사용된다. 즉, 기능의 확장과 재사용을 목적으로 한다.

- 자식 클래스에서 추상 메서드의 구현이 강제된다. (일반 메서드는 x)

- 상속하는 집합간에 연관관계가 있다.

*인터페이스란?

- 모든 멤버 함수와 프로퍼티가 추상 형태이다.

- 일반 멤버함수, 멤버변수, 프로퍼티를 포함할 수 없다.​

- 인터페이스는 다중 상속이 가능하다.

- IS-ABLE-To 관계에서 사용된다. 즉, 공통 기능의 부여를 목적으로 한다.

- 자식 클래스에서 구현이 강제된다.

- 상속하는 집합간에 연관관계가 없다. 클라이언트를 기준으로 인터페이스를 분리하기 때문.(아래에서 나올 '인터페이스 분리 법칙'에 의거)

​

​

(추상 클래스의 예)

public abstract class SomeClass{ public abstract int Something { get; set;} //추상 프로퍼티 public abstract void SomeFunc(int param); //추상 멤버함수 public int someThing2 = 3; //일반 멤버 변수를 포함할 수 있다. public int SomeThing3 { //일반 프로퍼티도 포함할 수 있다. get{ return someThing2; } set{ someThing2 = value; } } public void SomeFunc2(int param){ //일반 멤버 함수도 포함할 수 있다. Console.WriteLine("Hello!"); } }

(인터페이스의 예)

public interface SomeInterface{ public void SomeFunc(); //가상 함수 public int SomeProp{get; set;} //가상 프로퍼티 }

​

4. 다형성 (오버로딩, 오버라이딩)

오버라이딩: 부모 클래스의 타입으로 다루되, 자식 클래스에서 오버라이딩 했을 경우 해당 함수로 실행되게 하는 것. 부모 클래스와 자식 클래스가 둘다 독립적인 객체로 사용된다. 상속과 직접적으로 연관되어 있다.

오버로딩: 같은 이름의, 인자만 다른 함수를 여러 개 만드는 것. 상속과는 관련이 없음.

​

객체 지향적 설계 원칙

1. 단일 책임 원칙

클래스는 단 하나의 책임을 가져야 하며, 클래스를 변경하는 이유는 단 하나의 이유여야 한다.

책임이란, '클래스를 변경하려는 이유'로 정의한다.

ex) 보고서를 편집하고 출력하는 모듈의 경우, 보고서의 내용이 변하거나 형식이 변하면 모듈을 변경해야 한다. 따라서 '편집'과 '출력' 각각을 분리된 클래스로 나누어야 한다.

2. 개방-폐쇄 원칙

클래스는 확장에는 열려있어야 하고 변경에는 닫혀 있어야 한다.

기능의 확장 또는 변경이 있을 때 기존의 코드를 수정하지 않아야 한다. 즉, 새 모듈을 추가해도 해당 모듈을 사용하기 위해 기존의 코드를 건드리지 않아도 되도록 설계해야 한다.

3. 리스코프 치환 법칙

상위 타입의 객체를 하위 타입의 객체로 치환해도 상위 타입을 사용하는 프로그램은 정상적으로 동작해야 한다.

하위 타입의 객체는 상위 타입의 객체+@이므로, 상위 타입처럼 사용해도 문제가 발생하지 않아야 한다.

4. 인터페이스 분리 원칙

인터페이스는 그 인터페이스를 사용하는 클라이언트를 기준으로 분리해야 한다.

5. 의존 역전 원칙

고수준 모듈(인터페이스)은 저수준 모듈(객체)의 구현에 의존해서는 안 된다.

ex) 자동차와 스노우 타이어. 자동차는 고수준 모듈이고 스노우 타이어는 저수준 모듈인데, 스노우 타이어는 자동차에 의존해서 구현해도 되지만 자동차는 스노우 타이어에 의존해서 구현하면 안 된다. "타이어"라는 인터페이스를 만들어 자동차가 사용하게 해야 한다.

* 이 부분이 좀 헷갈리는데, 자동차는 공통 부분을 추출해 추상화한 인터페이스고, 스노우 타이어는 (아마도) 타이어 라는 인터페이스를 상속해 세부화한 객체일 것이다. 그런데 자동차에서 '타이어'가 아니라 '스노우 타이어'를 다루는 코드를 작성해버리면 '스노우 타이어'가 아닌 다른 종류의 타이어를 사용할 때마다 코드를 바꿔줘야 한다.

* 쉽게 말해, 비슷한 다른 객체로 교체할 가능성이 있는 객체를 다룰 땐 절대 그 특정 객체에 의존성을 만들지 말고 공통 부분을 추출해 인터페이스를 만든 뒤 그것에 의존해야 한다. 이는 개방-폐쇄 원칙과도 연결된다.

* 인터페이스가 객체보다 고수준 모듈이므로, 인터페이스가 인터페이스를 의존하거나, 객체가 인터페이스를 의존하는 것은 가능하나, 인터페이스가 객체를 의존하면 의존이 역전된다는 소리다.

** 참고로 이땐 다형성이 아니라 추상화를 사용하는 것이다.

​

** 다형성과 추상화의 차이점에 관한 고찰

다형성의 경우, 오버로딩을 사용하면 호출부의 인자를 다르게 해야 하므로 고수준 모듈의 코드에 수정이 가해져야 하고, 오버라이딩의 경우 인자가 필요하면 마찬가지로 고수준 모듈의 코드에 수정이 가해져야 한다. 오버라이딩은 함수의 내용은 바꿀 수 있지만 '추상적인 무언가'를 전달할 수는 없기 때문. 이 '추상적인 무언가'는 공통 부분을 추출한 부모 함수 또는 인터페이스가 된다.

결국, 다형성은 (1) 같은 부모 클래스를 상속받은 자식 클래스를 부모 타입으로 다루되 자식 클래스의 구현을 달리하여 실제 호출은 자식 클래스의 함수가 되게 하는 것이거나 (2) 같은 이름의 함수를 인자만 다른 형태로 호출하는 것이고, 추상화는 (1) 여러 클래스들에서 공통적으로 사용할 멤버만 인터페이스 또는 추상 클래스로 추출해, 호출부의 동작을 통일시키고 구현은 각 클래스에 맡기는 것이다.

​

다형성의 '오버라이딩'과 추상화의 차이점이 계속 헷갈렸는데, 다형성의' 오버라이딩'은 부모 클래스의 함수를 자식 클래스가 오버라이딩 했을 경우 해당 함수가 호출되지만 일단 부모 클래스 자체도 하나의 독립적인 객체로 사용된다는 점이 가장 큰 차이점인 것 같다. 추상화의 경우 부모 클래스는 인터페이스 또는 추상 클래스가 되는데, 인터페이스와 추상 클래스는 독립적인 객체로 사용될 수 없다.
