몬스터 클래스를 만들던 중 문제가 생겼다.


네모들이 중심 위치를 기준으로 좌우로 반복해서 움직이도록 만들었는데 창을 옮기거나 창의 크기를 조절하면 확률적으로 이상현상이 발생함.
네모들이 제자리에서 움찔거림.

창을 옮기거나 크기를 조절한다고 무조건 저렇게 되는 것도 아니고, 모든 네모가 항상 정지하는 것도 아니고, 완전히 정지하는 것도 아니라 제자리에서 움찔거림.

삽질 끝에 원인을 찾음.

```c++
void Monster::Update()
{
	//매 프레임 center-100에 도달할 때까지 pos.x-= move_speed_*dt
	//도달하면 center+100에 도달할 때까지 posx+= move_speed_*dt

	Vector2 pos = get_pos();
	if (abs(center_pos_.x - pos.x) > move_range_) {
		direction_ *= -1;
	}
	
	pos.x += direction_ * move_speed_ * Time::GetInstance()->dt_f();
	set_pos(pos);
	
}
```

### 원인

몬스터 클래스는 현재 위치인 pos와 중심 위치 center_pos, 그리고 이동 반경 move_range, 이동 속도인 move_speed를 가지고 있음.

중심 위치에서 현재 위치가 move_range보다 멀어지면 방향을 전환하도록 한 건데
문제는 왼쪽으로 멀어지든 오른쪽으로 멀어지든 구분하지 않고 무조건 방향을 반대로 바꾸는 것에 있었음.

가령, (50,50)에서 좌우로 10만큼만 움직이게 했는데 만약 모종의 이유로 네모가 오른쪽으로 너무 많이 벗어나서 (100000, 50)에 있다면
if문에서 방향은 왼쪽으로 전환됨.
해당 프레임의 이동 거리가 1이라면 왼쪽으로 1만큼 이동해서 (99999, 50)가 된 상태로 set_pos가 실행됨.
그리고 다음 프레임의 if문에서 마찬가지로 범위를 벗어나있는 것만 판단하고 방향을 또 오른쪽으로 바꿔버림.

즉, 이 네모는 x좌표를 40부터 60까지를 왔다갔다 해야 하는데, x좌표가 한 번의 프레임만에 돌아오지 못할 정도로 멀리 있다면 그때부터 제자리에서 미친듯이 요동치게 된다는 것.

### dt를 이동거리에 곱해줄 때 주의점

매 프레임의 이동거리는 이동속도*dt로 결정되는데, 이동속도는 고정값이므로 dt의 영향을 받음.
dt는 프레임당 걸린 시간인데,
하드웨어가 느려지면 값이 커지고, 하드웨어가 빨라지면 값이 작아짐. 따라서 모든 컴퓨터에서 일정한 속도값을 얻기 위해 사용됨.

문제는 dt는 (당연히) 매 프레임마다 바뀜.
따라서 저 코드는 모종의 이유로 dt가 확 커지면 그대로 에러현상에 뒤덮이게 되는, 정말 시한폭탄 같은 코드였던 것임.


dt는 그냥 컴퓨터가 잠시 렉 먹어도 확 커지지만 저 문제에선 윈도우를 움직일 때 발생했음.
뭘까 생각하니 지금 프로그램의 메인 루틴이

```c++
do
    {   
        if (PeekMessage(&msg, nullptr, 0, 0, TRUE)) {
            if (!TranslateAccelerator(msg.hwnd, hAccelTable, &msg))
            {
                TranslateMessage(&msg);
                DispatchMessage(&msg);

            }
        }
        else {
            Core::GetInstance()->Progress();
        }
        
    } while (msg.message != WM_QUIT);
```
이렇게 되어있는데 윈도우 메시지가 없을 때에만 게임의 메인 루틴을 실행하도록 함.
그리고 dt는 Core 안에서 계산하고 있음.
즉, 게임의 메인 루틴에 들어오지 못하면 dt는 하드웨어가 느려진 줄만 알고 왕창 커져버림.

윈도우를 움직이거나 크기를 변경할 때 발생하는 윈도우 메시지는 WM_MOVE와 WM_SIZE라고 함.
중단점 걸고 디버깅해보니 맞음.

그말인즉 게임을 하다가 윈도우 메시지가 대량 발생할 때마다 dt가 왕창 커진단 소리고 dt를 곱하는 모든 동작은 그때마다 무진장 큰 값을 뱉을 거란 소리임.

이걸 해결할 방법을 찾아야함.


일단 저 문제로 돌아가서 발생한 현상은 세 개임
1. 윈도우 메시지가 발생할 때마다 dt가 왕창 커짐
2. dt가 왕창 커질 때마다 이동거리가 너무 커짐
3. 이동 반경에서 크게 벗어나면 반경 안으로 돌아오지 못하고 요동침.

### 해결방법 

1번은 일단 해결 불가함. dt란 것의 개념이 원래 게임의 다음 프레임까지 얼마나 걸렸는지 체크하는 거기 때문.

2번은 min 함수로 해결할 수 있을듯함.
기본적으로 dt를 곱한 만큼 움직이지만 최댓값을 제한해둬서 이동속도가 너무 빨라지지 못하게 하는 것.

3번은 이동반경에서 벗어나면 이동반경의 가장자리로 위치를 바로잡아주면 됨.

```c++
void Monster::Update()
{
	//매 프레임 center-100에 도달할 때까지 pos.x-= move_speed_*dt
	//도달하면 center+100에 도달할 때까지 posx+= move_speed_*dt

	Vector2 pos = get_pos();
	if (abs(center_pos_.x - pos.x) > move_range_) {
		//가장자리로 바로잡기
		pos.x = center_pos_.x + direction_ * move_range_;
		direction_ *= -1;
	}
	
	pos.x += direction_ * min(move_speed_ * Time::GetInstance()->dt_f(), move_speed_);
	set_pos(pos);
	
}
```

이제 아무리 빨라져도 move_speed_ 이상으로 움직이지 못하고,
가장자리를 넘어가면 가장자리로 위치를 바로잡아줌.

테스트해보니 창을 클릭하고 옮기고 크기를 바꿔도 멀쩡하게 잘 돌아감.


근데 여기서 한가지 문제가 더 있음.
아까 멈추는 네모가 부분적으로만 발생했는데, 모든 몬스터는 한 프레임에서 동일하게 Update와 Render가 진행되므로 만약 dt가 너무 커졌더래도 모두 같은 dt로 연산을 하게 됨.
즉 에러가 똑같이 발생하거나 발생하지 않아야 맞는데 네모마다 오차가 있음.


### 부동소수점 오차 누적

바로 float형 변수끼리 덧셈을 할 때마다 오차가 조금씩 발생해서, 한 프레임에서 같은 값으로 입력을 받아도 연산 결과가 달라지는 거였음.
특히나 이 오차는 점점 누적되어서 프로그램이 오래 돌아갈수록 커지게 됨.

말로만 들어봤지 직접 겪는 건 처음임. 부동소수점 오차란 걸 배워서 뭔지 알고 있는데도 이 에러가 그것 때문일 거란 생각을 못해서 한참 삽질함.

```c++
pos.x += direction_ * min(move_speed_ * Time::GetInstance()->dt_f(), move_speed_);
```
지금 이 코드에서도 pos.x, move_speed_, dt_f(), move_speed_ 모두 float형 변수임.
move_speed_에 dt_f()를 곱하는 부분과 pos.x에 최종 연산 결과를 더하는 연산에서 오차가 발생하고 있을 것임.

디버깅 찍어보니까 모든 몬스터가 같은 move_speed_, dt, move_speed_를 사용하지만 pos.x에 +=을 하고난 뒤 값을 비교하면 다 조금씩 달랐음.
float은 6자리까지는 정확한데, 그 아래는 보장을 못함. 1.00000001 + 1.00000001을 한다면 매번 할 때마다 어떨 땐 2.000000142202 어떨 땐 2.00000001248 뭐 이런 식으로 나온단 소리임.

근데 이게 0.1234560001214 이런 값이 들어있어서 6자리 아래를 버리고 0.123456만 저장해도 다시 값을 찍어보면 0.123456003291 이런 값이 들어가있음.
거기다 소수점 아래 6자리가 아니라 123456000.0 같은 값도 맨 앞 6자리만 유효한 값이고 아래는 쓰레기값으로 생각해야 한다는 것임.

와...이건 int를 사용하도록 바꾸는 것이 아니면 근본적으로 해결할 방법이 없어보임.
일단은... 누적 덧셈이 이루어지는 곳이

```c++
pos.x += direction_ * min(move_speed_ * Time::GetInstance()->dt_f(), move_speed_);
```
이 부분이고 여기서 문제가 발생하는 것이니까

direction_은 1또는 -1
move_speed는 1~0.001 정도의 범위에서만 왔다갔다 할 듯한 소수
dt_f는 0.1~0.0001 정도에서 왔다갔다할 소수

이걸 고려하면 왼쪽으로 자리수를 올려주고 int형으로 변환해 아랫값을 버린 뒤 연산하고 자리수를 다시 내려주면 될 듯함.

그리고 메모리 때문에 가능하면 double보다 float을 쓰라고 하는데 float은 6자리(-999만~+999만)만 저장할 작은 범위의 숫자를 다룰 때만 쓰도록 하고
큰 숫자를 다룰 땐 double을 쓰는 게 좋을 것 같음.

근데 double도 float보다 오차의 값이 아주 작은 오차가 항상 생기기 때문에 누적될 경우 언젠가 미래의 어떤 시점부터 슬슬 오차가 눈에 보이는 결과로 나타나기 시작할 것임.

변수에 들어있는 값의 첫번째 자리수를 구한 다음 그로부터 6번째 자리수부터 잘라서 int형 정수로 잘라버리는 방법도 있겠지만
매번 이 처리를 해주고 연산이 끝나면 되돌려주고...
너무 고된 작업임. 덧셈은 전부 올려주고 더한 다음 전부 같은 자리수로 내려주면 되지만, 곱셈이나 나눗셈의 경우 실수할 가능성이 높고.
두 수의 자리수가 다르다면 (ex: 1234560000.0 + 0.000123456 ) 또 문제가 발생함.

일단 몬스터의 동작이 무조건 똑같아야 한다는 요구조건은 없으므로 그냥 이대로 놔두려 함.
그리고 가장자리에 닿으면 다시 바로잡아지기 때문에 앞으론 오차가 쌓이지 않을 것이기도 하고...

부동소수점 오차 문제는 완전한 해결법이 없어서 계속 생각해봐야할듯함.

아무튼 결론.

1. 부동 소수점을 쓸 땐 가능하면 double을 사용하자. (요즘은 메모리 절약의 중요성이 과거보다 떨어졌으므로 double을 쓰는 것도 나쁘지 않은 방법 같음.)
2. 연산시엔 double로 형변환해서 연산하거나, 소수점의 범위를 알고 있다면 자리수를 올리고 int형으로 바꾸어 연산 후 자리수를 내리는 방법을 쓰자.
3. 부동 소수점 비교 연산은 ==을 사용하면 안 된다. abs(a-b) < FLT_EPSILON을 사용하자. (double이면 DBL_EPSILON)
4. 부동 소수점 오차가 누적되는 연산은(a += b 등) 오차가 누적되면 어떤 영향이 올지 반드시 고려하고 해결법을 적용할 것.
